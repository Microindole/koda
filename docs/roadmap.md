# koda 项目深度技术路线与演进纲领

## 1. 项目愿景与定位

koda 定位于一个高响应性的“私有感官终端”。它通过 ArkTS 捕获物理世界信号，利用 HarmonyOS 原生声明式 UI 能量，将枯燥的传感器数据转化为具有极客美学与二次元氛围的交互艺术。项目坚持本地化、轻量化与模块化，作为探索 ArkUI 极限性能的实验平台。

## 2. 核心技术栈

- 开发语言：ArkTS (Strict Mode) 强制类型约束，确保逻辑严密。
- UI 框架：ArkUI 声明式开发，通过属性动画（Attribute Animation）与显式动画（Explicit Animation）实现视觉特效。
- 环境感知：`@ohos.sensor` 核心库，重点突破光照（Ambient Light）与运动（Pedometer/Accelerometer）的数据采集。
- 数据持久化：`@ohos.data.relationalStore` 用于结构化历史记录，`@ohos.data.preferences` 用于用户偏好配置。
- 并发处理：`@ohos.taskpool` 负责高频率传感器数据的清洗与计算，避免阻塞 UI 主线程。
- 触觉反馈：`@ohos.vibrator` 实现精细化的震动反馈。

## 3. 架构演进设计

### 3.1 感知抽象层

- 建立统一的 `VibeProvider` 接口，将不同类型的传感器数据标准化。
- 实现数据预处理管道：包括去噪算法（Noise Reduction）与灵敏度调节，确保输出数据在 0.0 到 1.0 之间归一化。

### 3.2 逻辑分发层

- 采用单一数据源状态管理。建立 `kodaState` 全局对象，通过装饰器驱动 UI 自动刷新。
- 实现环境模式识别：基于光照与运动频率，自动在 `Stealth` (暗色极客)、`Vivid` (高饱和二次元)、`PowerSave` (极简低功耗) 模式间切换。

### 3.3 表现层组件化

- 放弃 Canvas 绘图，全面采用声明式组件堆叠。
- 利用 `Stack` 容器构建多层背景：底色层、纹理层、动态数据层。
- 广泛应用 `transition` 与 `animation` 属性，实现流畅的非线性交互效果。

## 4. 阶段性里程碑

### 第一阶段：感官基石

- 建立 `core/perception` 模块，封装光照与步数监听器。
- 解决本地模拟器环境下的传感器数据联调，确保数据能稳定驱动页面变量。
- 技术重点：传感器权限处理与 `aboutToAppear/aboutToDisappear` 生命周期内的资源回收。

### 第二阶段：声明式视觉协议

- 确立 koda 的视觉基调：基于 One Dark 的深色背景搭配 B 站粉 (#FB7299) 与极客绿 (#00FF00)。
- 实现“光感氛围系统”：背景明暗与组件色散随 lux 值动态演变。
- 技术重点：响应式布局布局适配与组件级属性动画的性能调优。

### 第三阶段：多模态交互反馈

- 引入触觉维度：为步数增加脉冲震动，为极端光照变化增加预警震动。
- 实现“音画同步”实验：监听本地音频波动，驱动组件形状或缩放产生律动。
- 技术重点：精密震动波形设计与资源占用平衡。

### 第四阶段：离线数据实验室

- 实现 RDB 数据库接入，记录每日感官数据波动。
- 开发“时空回溯”功能：可视化展示过去 24 小内的环境变化趋势。
- 技术重点：数据库异步操作封装与大规模数据加载优化。

### 第五阶段：分布式协同

- 探索分布式流转：实现跨设备的环境状态同步。
- 建立“协同感官”：手机作为传感器源，平板作为视觉加强终端。
- 技术重点：分布式软总线发现机制与跨端数据对象同步。

## 5. 性能与合规规范

- 渲染性能：所有复杂动画均需开启硬件加速，确保在 120Hz 屏幕下不掉帧。
- 隐私策略：应用坚持零网络权限，所有物理世界数据严格限制在设备本地存储。
- 资源消耗：严格控制后台常驻进程，传感器监听在应用挂起时必须进入休眠模式。

## 6. 未来扩展性

架构设计需兼容未来可能加入的“看板娘实体”或“外部生物识别模组”。所有表现层组件必须支持动态插槽，确保在不改动核心感知逻辑的前提下，能够通过更换样式包实现完全不同的 UI 视觉体验。
