# Agent 标准工作流

## 0. 硬约束

- 必须遵守 `.agent/AGENTS.md`：
  - 中文输出。
  - 发现问题要直接指出，不回避。
  - 未完成项加 `TODO`，已知未修复缺陷加 `FIXME`。

## 1. 接单与澄清

1. 复述任务目标（功能、范围、输出物）。
2. 明确边界：
   - 是否允许改动目录结构。
   - 是否要求测试/截图/性能数据。
3. 如果信息不足，先做最小可行假设并显式写出。

## 2. 上下文收集

1. 先读：
   - `.agent/PROJECT_BRIEF.md`
   - `docs/roadmap.md`
2. 再读直接相关代码文件，避免全仓扫描。
3. 识别变更影响面：
   - 入口文件。
   - 状态流。
   - 资源与配置。

## 3. 实施策略

1. 优先小步提交：
   - 先搭骨架，再填实现，再补测试。
2. 约束新增复杂度：
   - 先接口，后细节；先可运行，后优化。
3. 对临时方案显式标记：
   - `TODO`：计划后续完成。
   - `FIXME`：已识别问题但本次未解。

## 4. 验证

1. 最少验证：
   - 编译通过。
   - 受影响测试通过。
2. 若无法验证，必须说明：
   - 缺少的环境。
   - 未验证风险点。

## 5. 交付

1. 先给结果，再给细节。
2. 必须包含：
   - 改了哪些文件。
   - 为什么这样改。
   - 还剩哪些风险/TODO/FIXME。
3. 若有后续动作，给 1~3 条可执行下一步。
